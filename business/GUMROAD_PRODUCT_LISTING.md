## Short description

## Full product description

This product is the Professional Edition of **KRITERION**.

It is designed for individual consultants and small teams that need a defensible way to evaluate security capability without falling back on résumé theater, interview fluency, or manager narrative.

What you get is not a template pack. It is a structured framework.

The package includes:
- five role protocols covering Security Engineer, Senior / Lead Security Engineer, Enterprise / Lead Security Architect, Principal Security Engineer, and Distinguished Security Engineer;
- the GPT5.4 Audit Hardening Protocol that makes the evaluation path deterministic and LLM-safe;
- machine-readable schemas for canonical artifacts, evaluation results, gates, and handoffs;
- execution documents covering deterministic run behavior, failure codes, orchestration, and canonicalization;
- Python reference tools for validation and reference execution;
- a worked example with trace output;
- a synthetic benchmark harness that demonstrates the repository’s runnable evaluation loop;
- doctrine documents that explain the design logic behind the framework.

The problem this solves is simple: most evaluation systems fail open.

When evidence is weak, people compensate with confidence.
When artifacts are incomplete, evaluators improvise.
When the candidate communicates well, the score drifts upward.
When promotion evidence is self-referential, the loop often goes undetected.

This product corrects that pattern.

### Scenario 1 — Candidate evaluation
You are assessing an external candidate for a security engineering role. The candidate sounds strong, but the artifact bundle is uneven. The protocols force the evaluation to separate communication quality from evidence quality, apply confidence caps, and block unjustified score inflation.

### Scenario 2 — Internal promotion
You are reviewing a promotion packet where the same work is being cited repeatedly across domains and the approval chain is too close to the author. The anti-gaming and independence logic helps you distinguish real breadth from organizational over-crediting.

### Scenario 3 — AI agent grounding
You want an AI-assisted evaluation workflow that does not let repo text, comments, or artifacts silently override system logic. The hardening layer defines how artifacts are normalized, validated, and treated as non-authoritative data.

Why this does not already exist in a usable form:
existing materials tend to split into two camps. One camp provides broad security or audit frameworks that are not built for human role-ladder evaluation. The other provides rubric-based AI evaluation patterns that do not treat evidence admissibility, reviewer independence, prompt injection, and anti-gaming as first-class mechanisms. This package combines those concerns in one execution-grade object.
